# 会計処理練習ツール 実装指示書（Codex用）

## 0. 文書位置づけ

本書は、会計処理練習ツールの **実装指示書** である。  
本書は、「どう作るか」「何をしてはならないか」「どの順で作るか」を定義する。  
本書は、別紙の **仕様書** を前提とし、仕様書に優先して機能追加・機能削除を行ってはならない。  
仕様書に明記されていない判断を、Codexが勝手に補完してはならない。  
Codexは `AGENTS.md` を作業前に読む運用が公式に案内されているため、本書は `AGENTS.md` または同等の実行指示文書として扱う。 :contentReference[oaicite:1]{index=1}

### 0.1 最上位拘束
- 仕様書が **何を作るか** の唯一の基準である。
- 本書が **どう作るか** の唯一の基準である。
- 仕様書と本書の両方に書かれていない事項は、実装対象に含めない。
- 不明点があっても、仕様範囲を広げる方向で補完してはならない。
- 実装時の判断は、**MVPを小さく保つ方向** を優先する。
- Codexはコード読取・編集・実行を進められるが、作業可能であることと実装許可は同義ではない。許可範囲は本書で拘束する。 :contentReference[oaicite:2]{index=2}

### 0.2 成果物の定義
Codexが今回作成してよい成果物は、以下に限定する。

- アプリ本体コード
- 必須設定ファイル
- 必須依存関係定義
- 必須DBマイグレーション
- 最小限の起動手順書
- 最小限のテストコード
- 実装進捗メモ

Codexが今回作成してはならない成果物は、以下とする。

- 仕様書の再解釈版
- 未承認の追加機能
- 過度に高度な管理画面
- MVP対象外機能の先行実装
- 税務申告機能
- 提出書類作成機能
- 自動論点推定
- 自動再構成支援
- 補助簿群の詳細実装
- 原価計算・工業簿記の専用実装
- 単式簿記モードの実装

## 1. 実装の最優先原則

### 1.1 仕様優先
仕様書に書かれた内容のみ実装対象とする。  
仕様書にない利便機能を追加してはならない。

### 1.2 MVP優先
第15章のMVP範囲だけを実装対象とする。  
MVP対象外機能は、設計の痕跡を残すことは許可するが、動作実装してはならない。

### 1.3 差分最小化
実装は、最小差分・最小構成・最小依存関係で行う。  
将来拡張のために過剰設計してはならない。

### 1.4 再現性優先
ローカルで再現可能な構成を優先する。  
実装・起動・DB初期化・テストの手順は、最小限で再現できる状態にする。

### 1.5 監査可能性優先
重要データは物理削除せず、状態管理で無効化する。  
回答履歴と採点結果は分離する。  
過去問更新時は改訂履歴を保持する。

## 2. Codexの行動制約

### 2.1 勝手にやってはならないこと
Codexは、以下を行ってはならない。

- MVP対象外機能の先行実装
- 仕様にない画面の追加
- 仕様にないテーブルの増設
- 仕様にない採点ロジックの追加
- 仕様にない税金論点の拡張
- 仕様にない権限管理の追加
- UIの全面的な独自アレンジ
- 高度な認証基盤の導入
- マイクロサービス分割
- 外部SaaS前提の構成
- 本番運用前提のインフラ構成
- 過去問の著作権区分未設定での出題許可
- 原文収録不可問題の原文表示保持

### 2.2 補完ルール
Codexが補完してよいのは、以下に限る。

- 命名規則の統一
- フォルダ構成の自然な整理
- 既存仕様に沿ったバリデーション追加
- 既存仕様に沿った型定義追加
- 既存仕様に沿ったエラーメッセージ整備
- 既存仕様に沿ったテスト追加
- 既存仕様に沿ったUIラベルの自然な調整

### 2.3 不明点の扱い
不明点がある場合、Codexは以下の順で処理する。

1. 仕様書を再確認する  
2. 本書を再確認する  
3. それでも不明であれば、**実装を広げずに保守的に止める**  
4. 仮実装する場合は、拡張ではなく最小安全側を選ぶ

## 3. 技術方針

### 3.1 推奨実装形態
MVPは **単一Webアプリ** として実装する。  
フロントエンドとバックエンドを同一リポジトリで管理する。  
DBはローカルで容易に起動・初期化できる構成を採用する。

### 3.2 設計方針
- 単一リポジトリ
- 単一アプリケーション
- 単一DB
- API層と画面層は責務分離
- 採点ロジックはサービス層へ分離
- 過去問取込はバッチ処理として分離
- 帳簿影響表示はテンプレートデータ参照方式とする

### 3.3 可変データの扱い
MVPでは、問題種別ごとの可変入力は JSON 等の構造化データで保持する。  
問題種別ごとの専用回答テーブルは作らない。  
安定後の第2段階で必要なら分解する。

### 3.4 設定ファイル
- 依存関係定義ファイルを明示する
- 環境変数サンプルを用意する
- 開発用設定と本番用設定を無理に分けない
- MVPではローカル起動優先とする

### 6.1 必須テーブル
MVPで必須のテーブルは、仕様書 9.6 の一覧に限定する。

### 6.2 テーブル追加禁止
以下はMVPで追加してはならない。

- 単式簿記専用回答テーブル
- 原価計算専用回答テーブル
- 工業簿記専用回答テーブル

- 高度監査ログ専用テーブル
- 高度権限管理専用テーブル

### 6.3 状態管理
- 物理削除禁止
- retired / suspended / draft 等で管理
- 回答も履歴保持
- 採点再実行が可能な構造を維持

### 6.4 データ整合ルール
- answer と score_result は分離
- acceptable_answers は複数保持可能
- question_parts は総合問題用に必須
- past_exam_import_batches と past_exam_import_errors は分離
- account_master に account_description を必ず持たせる

## 7. 採点エンジン実装指示

### 7.1 必須採点対象
MVPでは以下のみ必須実装とする。

- 複式簿記問題
- 税金仕訳問題
- 基本的な帳簿転記問題
- 過去問ベース小問

### 7.2 採点ルール固定
- 100点満点の要素配点方式
- 学習モードと試験モードの分離
- 表記ゆれ吸収
- 必須語句方式
- 税金仕訳の前提一致判定
- 行単位部分点
- score_result_details 保存

### 7.3 後回しにする採点
MVPでは以下を本実装しない。

- 高度な理論記述採点
- 高度な同義語辞書拡張
- 自動解説生成
- 試験回別最適化採点
- 原価計算・工業簿記の本格採点

## 8. 過去問管理実装指示

### 8.1 必須実装
- 手動登録
- CSV一括登録
- 行単位エラー保存
- 重複候補検出
- 問題状態管理
- 改訂履歴保存
- 著作権区分必須化
- 出題可否制御
- 独自再構成問題への切替

### 8.2 著作権区分ルール
- 未設定では登録完了不可
- 利用条件未確認は出題不可
- 収録不可は保存可能だが出題不可
- ユーザー私有教材は本人のみ出題可能
- 原文収録不可時は独自再構成問題を優先使用

### 8.3 禁止事項
- 原文不可問題の原文保持
- 著作権区分未設定の仮公開
- publishable 以外の出題
- 物理削除

## 11. 完了条件

### 11.1 実装完了条件
Codexは、以下を満たしたときのみ「MVP実装完了」と扱ってよい。

- 仕様書第15章のMVP範囲に含まれるものだけを実装した
- 仕様書第16章の受け入れ基準を満たす見込みがある
- 起動手順が成立する
- DB初期化が成立する
- 主要画面が遷移する
- 回答、採点、保存、履歴、過去問登録が成立する
- 著作権制御が成立する
- 明らかなMVP外機能が混入していない

### 11.2 完了報告形式
Codexの完了報告は、以下の順でまとめる。

- 実装した範囲
- 実装していない範囲
- 仕様どおりに拘束した点
- テスト結果
- 残課題
- 起動方法

## 12. 進行メモ更新ルール

### 12.1 必須記録
Codexは、作業中に以下を更新してよい。

- 現在の実装段階
- 完了した実装単位
- 未完了の実装単位
- 仕様上の未解釈箇所がなかったこと
- テスト結果
- 既知不具合

### 12.2 禁止記録
Codexは、以下を記録してはならない。

- 仕様変更提案を既成事実として書くこと
- 未承認の機能追加予定を書くこと
- 「ついでに追加した」機能を正当化すること

## 13. Codexへの最終命令

- 仕様書に従ってMVPのみを実装せよ。
- 本書に従って実装順・禁止事項・保存方針を守れ。
- 仕様書にない判断を勝手に補完するな。
- MVP対象外機能を実装するな。
- 差分は小さく保て。
- 主要動作とデータ整合を優先せよ。
- 受け入れ基準に直結するものから順に完成させよ。
- 完了時は、実装範囲と未実装範囲を明確に報告せよ。
